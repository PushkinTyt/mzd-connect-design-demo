Архитектура IPCIPC Architecture

CMU использует два транспорта: WebSocket для GUI ↔ MMUI и DBus для взаимодействия между нативными сервисами. Все сервисы управляются Service Manager (sm_svclauncher).

CMU uses two transports: WebSocket for GUI ↔ MMUI and DBus for native service IPC. All services are managed by the Service Manager (sm_svclauncher).

Diagnostics service. Binary: svcjcitestdiag.so. Receives requests from MMUI (via WebSocket) and translates them into diagnostic operations via DBus.

com.jci.UPDATEA_Sys_Ipc.Interface

Firmware update agent. Binary: svcjciupdatea.so. Searches for update packages (.up) on USB, verifies signature, installs, then initiates reboot via BLM_SYSTEM.

com.jci.guiManager — Opera Watchdog

Opera JS-плагин (guimanager.so). Следит за работоспособностью GUI: если JavaScript не присылает heartbeat в течение 10 секунд — процесс Opera принудительно завершается и перезапускается.

Opera JS plugin (guimanager.so). Monitors GUI health: if JavaScript stops sending heartbeats for 10 seconds, the Opera process is killed and restarted.

VBS — Vehicle Bus Services

Набор сервисов-мостов между HMI и TAU (Tuner/Audio Unit). Транслируют DBus-вызовы в команды для физических модулей (тюнер, усилитель, CAN-шина автомобиля). Бинари используют единый объектный слой JCIDBUS.

Bridge services between HMI and TAU (Tuner/Audio Unit). Translate DBus calls into commands for physical modules (tuner, amplifier, vehicle CAN bus). Binaries share the common JCIDBUS object layer.

Three Lua scripts manage the full audio stack. Communicate via DBus (/tmp/dbus_service_socket and /tmp/dbus_hmi_socket). All started via xsesvc2dbus — bridge between XSE and DBus.
